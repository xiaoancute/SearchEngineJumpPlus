# 自动获取站点 Favicon 设计

**目标**

让所有内置搜索站点默认按对应网址自动获取 favicon，而不是长期依赖 `engineList.js` 里手写的图标地址；同时保留稳定回退，避免图标再次大面积空白。

**当前现状**

- 跳转栏和设置面板主要直接读取每个引擎对象上的 `favicon`
- 脚本已经有 `getICON(url)`，能按 `settingData.getIcon` 生成 favicon 地址
- 旧配置会把历史 `favicon` 缓存在本地，后续就算默认数据改了，界面也可能继续显示旧值

**设计方案**

## 1. 统一图标解析入口

新增一个统一的图标解析逻辑，所有内置站点在渲染时都优先按 `url` 自动获取 favicon：

- 如果是用户手动设置过的自定义图标，继续使用该值
- 否则按当前 `getIcon` 配置，通过 `getICON(url)` 自动生成图标地址
- 如果自动生成结果不可用，再回退到引擎对象当前自带的 `favicon`

这样可以让“自动获取 favicon”成为默认行为，但不破坏已有兜底能力。

## 2. 旧缓存迁移

在设置初始化阶段补一段迁移逻辑，只处理内置站点：

- 如果本地缓存里的内置站点 favicon 是历史遗留值，就更新为新的“自动获取优先”状态
- 不覆盖用户手动填写的自定义图标
- 让已经安装旧版本的用户无需手动清空油猴存储

## 3. 作用范围

这次只调整“内置搜索站点”：

- `engineList.js` 中自带分类和站点统一适配
- 用户后来在设置面板自行新增的站点，继续沿用现有逻辑

**实现边界**

- 不改搜索分类结构
- 不改搜索 URL 生成逻辑
- 不引入额外远程服务
- 不删除现有 `favicon` 字段，保留为回退

**验证方案**

1. 增加回归测试，验证脚本源码包含“统一图标解析”和“旧缓存迁移”逻辑
2. 用 `node --check` 验证脚本语法
3. 手工确认内置站点在跳转栏和设置面板都走同一套 favicon 解析策略

**风险与处理**

- 某些站点 `/favicon.ico` 可能 404、403 或重定向异常
  - 保留当前 `favicon` 作为回退
- 本地旧配置可能覆盖默认行为
  - 初始化时补迁移逻辑
- 用户可能已经手工换过图标
  - 迁移时仅处理内置站点的历史默认值，不改手工自定义值
